## ‚ö° The Problem

Microsoft updates the large language models powering your Copilot Studio bots **silently and without notice**. A model swap changes your bot's accuracy, tone, and topic coverage overnight. You have no visibility into when it happened, how much changed, or which test cases flipped.

You find out from a support ticket. Not a dashboard.

---

## ūüéĮ What ńĀshokńĀ does

**ńĀshokńĀ** is an autonomous agent that sits between your Copilot Studio bots and your team. It watches every bot you care about around the clock. The moment a trigger fires ‚ÄĒ a model version change, a manual force, or any event you wire in ‚ÄĒ ńĀshokńĀ:

1. **Fires the Copilot Studio Eval API** ‚ÄĒ discovers all test sets, triggers every one in parallel
2. **Scores every test case** ‚ÄĒ pass/fail, numeric scores, per-case AI reason text
3. **Compares against the previous run** ‚ÄĒ classifies each metric as REGRESSED, IMPROVED, or STABLE
4. **Consults an LLM** ‚ÄĒ uses web search + your bot's own system prompt to produce a plain-English analysis of what changed and why, distinguishing model effects from persistent agent weaknesses
5. **Reports** ‚ÄĒ persists everything locally, generates a self-contained HTML report, and emails it to you

All of this happens before your users notice anything.

---

## ūüß≠ Philosophy

> **ńĀshokńĀ observes. Humans decide.**

ńĀshokńĀ is a pure observer. It has no ability to roll back a model, modify a bot, or take corrective action. Its job is to surface the truth of what changed, with enough signal for a human to decide.

This is deliberate. Automated rollbacks of AI systems carry their own risks. ńĀshokńĀ gives your team the signal ‚ÄĒ the decision is always yours.

- No automated rollbacks or model changes
- No writes to Dataverse or Copilot Studio
- Pure, unobtrusive observation and structured reasoning

## ūüß† LLM Reasoning ‚ÄĒ ńĀshokńĀ as a guide

ńĀshokńĀ doesn't just score test cases. It consults an LLM to explain the delta in plain language ‚ÄĒ acting as an intelligent guide, not just a metrics dashboard.

Before calling the LLM, ńĀshokńĀ:
- **Searches the web** (via Tavily, if configured) for release notes and known capability differences between the old and new model
- **Injects the bot's own system prompt** so the LLM understands the bot's purpose and instructions
- **Sends the full per-case comparison** ‚ÄĒ old model reason vs new model reason for every test case

The LLM is asked to distinguish:
- **Model effects** ‚ÄĒ cases where the old model passed and the new model failed (or vice versa) ‚Üí these point to model capability changes
- **Persistent weaknesses** ‚ÄĒ cases where both models fail ‚Üí these are agent/prompt issues, not model issues

The output is a two-part narrative written for a mixed architect and business audience:

1. **Analysis** ‚ÄĒ what changed, why it changed, grounded exclusively in the case evidence. No assumptions about model capabilities are made; every claim must trace to a specific score or evaluator reason in the data.
2. **Best-practice layer** *(only when metrics regressed or improved)* ‚ÄĒ a second LLM call produces architect-level recommendations: system prompt changes, grounding strategy, test coverage gaps, rollback criteria, and stakeholder communication ‚ÄĒ all tied to the failure pattern identified in part one.

The LLM analysis is generated automatically on every eval run and stored in `run.json`. It is also available on demand from the **ask ńĀshokńĀ** panel on the bot detail page.

You can use any OpenAI-compatible endpoint: OpenAI, Azure OpenAI, Azure AI Foundry, or a local model.

### Step 2 ‚ÄĒ Create an Entra ID app registration

The agent uses **delegated auth** ‚ÄĒ it calls APIs as you, not as a service principal. Microsoft requires this for the Copilot Studio Eval API.

4. **Authentication** ‚Üí **Add a platform** ‚Üí **Mobile and desktop applications**
   ‚Üí tick `https://login.microsoftonline.com/common/oauth2/nativeclient` ‚Üí **Configure**
5. **API permissions** ‚Üí **Add a permission** ‚Üí **APIs my organization uses**
   ‚Üí search `Power Platform API` (GUID `8578e004-a5c6-46e7-913e-12f58912df43`)
6. **Delegated permissions** ‚Üí tick all three:

| Permission | Purpose |
|---|---|
| `CopilotStudio.MakerOperations.Read` | List test sets, retrieve eval results |
| `CopilotStudio.MakerOperations.ReadWrite` | Trigger a new evaluation run |
| `EnvironmentManagement.Environments.Read` | Auto-discover Power Platform environments |

7. **Add a permission** ‚Üí **APIs my organization uses** ‚Üí search `Dynamics CRM`
   (GUID `00000007-0000-0000-c000-000000000000`)
8. **Delegated permissions** ‚Üí tick `user_impersonation` ‚Üí **Add permissions**
9. **Grant admin consent for [tenant]** ‚Üí confirm

| Permission | Purpose |
|---|---|

> **Note:** Without `user_impersonation`, model version shows as `unknown` and change detection is skipped. Without `ReadWrite`, ńĀshokńĀ can read test sets but cannot trigger eval runs. Without `EnvironmentManagement.Environments.Read`, use manual environment entry in Setup.

## ūüďč Event log

## ūüĒź Auth model

All API calls ‚ÄĒ Copilot Studio Eval API, Power Platform environmentmanagement, Power Platform Inventory, Dataverse ‚ÄĒ share one MSAL `PublicClientApplication` and one `SerializableTokenCache` file. Device flow runs once via the Setup page; all subsequent calls acquire silently via the cached refresh token.

| Scenario | Behaviour |
|---|---|
| First run | MSAL device flow ‚ÄĒ sign in once via the Setup page; token cached to disk |
| Subsequent runs | Silent refresh via cached token ‚ÄĒ no user interaction |
| Token expires | Agent emails a new device code to the configured recipient; polls for 15 min |
| Docker / Azure | `msal_token_cache.json` lives in `$STORE_DIR/agent/` ‚ÄĒ shared volume covers both containers |

No secrets in environment variables. All credentials ‚ÄĒ LLM API key, SMTP password, Tavily key ‚ÄĒ are stored in `config.json` and written exclusively by the Setup page.

## ūü©ļ Troubleshooting

| Symptom | Fix |
|---|---|
| `0 bots found` | Rerun Setup ‚Üí Agents ‚Üí Load Bots ‚ÄĒ or check `monitoredBots` in `config.json` |
| `no test sets found` | Create a test set in Copilot Studio ‚Üí your bot ‚Üí Evaluation tab |
| Nothing in dashboard after setup | Use **‚Ė∂ Force Eval** on the bot detail page to queue an immediate run |
| Model version shows `unknown` | App registration is missing `Dynamics CRM ‚ÄĒ user_impersonation` delegated permission with admin consent |
| `model swap: unknown ‚Üí unknown` in LLM analysis | Same root cause ‚ÄĒ MSAL cannot exchange the refresh token for a Dataverse access token |
| M365 Copilot Agent Builder agents show `default` | Correct ‚ÄĒ these agents have no selectable model; ńĀshokńĀ skips Dataverse lookup |
| LLM 401 / 403 error | Check `llm.api_key` in `config.json` via the Setup page ‚ÄĒ key must match the endpoint |
| `MSAL auth failed` | Re-authenticate via Setup ‚Üí Authentication ‚Üí Sign In |
| 403 on Load Environments | App registration needs `EnvironmentManagement.Environments.Read` with admin consent |
| SMTP failed | Office 365: host `smtp.office365.com`, port `587`, TLS ‚ÄĒ check password in Setup ‚Üí Notifications |
| Container exits immediately | `docker compose logs ashoka-agent` ‚ÄĒ check for missing volume mount or invalid `config.json` |
| Timeline empty | Run a force eval from the bot detail page ‚ÄĒ it writes the first events to `data/agent/events.jsonl` |
| Logs tab empty | Start the agent ‚ÄĒ `data/agent/agent.log` is created on first run |
| LLM calls not visible in Logs | Set `log_level: "DEBUG"` in config ‚ÄĒ LLM and web API calls are at DEBUG level by default |
| Memory warning in log | Agent RSS grew >50% from baseline ‚ÄĒ check for large Dataverse or eval API payloads |
| Eval quota reached | Copilot Studio Eval API caps at ~20 evals per bot per 24 h ‚ÄĒ `error` is logged, next cycle retries |
| `ask ńĀshokńĀ` shows button but no analysis | Analysis is auto-persisted on every agent run ‚ÄĒ click the button to regenerate on demand if the cached text is missing |
